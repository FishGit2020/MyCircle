# Implementation Plan: Personal Notes

**Branch**: `008-personal-notes` | **Date**: 2026-03-20 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/008-personal-notes/spec.md`

## Summary

Enhance the existing `packages/notebook` MFE and `functions/src/resolvers/notes.ts` to add GraphQL write operations (add, update, delete mutations) and migrate the `useNotes` hook from the `window.__notebook` window-global pattern to Apollo mutations. The `notes` query already exists; read path is already GraphQL-first. This brings the notebook's write path into full compliance with Constitution Principle III while satisfying all 11 functional requirements from the spec.

**No new MFE package is created.** The existing `/notebook` route, nav item, shell integration, Dockerfile, dev scripts, and i18n keys are already in place.

## Technical Context

**Language/Version**: TypeScript 5.x
**Primary Dependencies**: React 18, `@mycircle/shared` (Apollo re-exports, i18n, WindowEvents, PageContent), `react-router`, Tailwind CSS
**Storage**: Firestore `users/{uid}/notes/{noteId}` — existing collection, no schema migration needed
**Testing**: Vitest, React Testing Library (`@testing-library/react`)
**Target Platform**: Web (Vite Module Federation, served via shell at `localhost:3000`)
**Project Type**: Enhancement to existing MFE (`packages/notebook`, port 3010)
**Performance Goals**: Search results visible within 1 second for ≤ 500 notes; mutations complete within 2 seconds on typical connection
**Constraints**: No new npm dependencies; no new MFE shell integration points; no changes to existing public-notes feature
**Scale/Scope**: Personal use — up to 500 notes per user; single Firestore sub-collection

## Implementation Phases

### Phase 1 — Backend: Schema + Resolver + Codegen

1. Add `NoteInput` and `NoteUpdateInput` input types to `functions/src/schema.ts`
2. Add `addNote`, `updateNote`, `deleteNote` mutations to the Mutation type
3. Implement mutations in `functions/src/resolvers/notes.ts`; increase default limit to 500
4. Register mutations in `functions/src/resolvers/index.ts`
5. Add `GET_NOTES`, `ADD_NOTE`, `UPDATE_NOTE`, `DELETE_NOTE` to `packages/shared/src/apollo/queries.ts`
6. Run `pnpm codegen` to regenerate `packages/shared/src/apollo/generated.ts`

### Phase 2 — Frontend: Hook Migration

7. Update `packages/notebook/src/hooks/useNotes.ts` to use `useQuery(GET_NOTES)` and `useMutation(ADD_NOTE / UPDATE_NOTE / DELETE_NOTE)` from `@mycircle/shared`
8. Ensure `saveNote(null, data)` → `ADD_NOTE` mutation; `saveNote(id, data)` → `UPDATE_NOTE` mutation
9. Ensure `deleteNote(id)` → `DELETE_NOTE` mutation with `refetchQueries`
10. Keep the real-time `api.subscribe` path if `window.__notebook.subscribe` is available (belt-and-suspenders for cross-device sync)

### Phase 3 — Frontend: Component Cleanup + i18n

11. Verify `NoteList.tsx` search input is wired to the hook correctly (client-side filter on the returned notes)
12. Verify `NoteEditor.tsx` Save button is disabled when both fields empty
13. Add any missing i18n keys (`notebook.untitled`, `notebook.noResults`) to all 3 locales
14. Run `pnpm build:shared`

### Phase 4 — Tests + Validation

15. Update `useNotes.test.ts` to mock Apollo mutations instead of `window.__notebook`
16. Run `pnpm --filter @mycircle/notebook test:run`
17. Run `pnpm lint && pnpm typecheck`
18. Run `cd functions && npx tsc --noEmit`
19. Commit and open PR

## Dependencies & Execution Order

```
Phase 1 (backend) must complete before Phase 2 (hook migration) —
the hook imports generated types that don't exist until pnpm codegen runs.

Phase 3 can run in parallel with parts of Phase 2 (i18n is independent of hook logic).

Phase 4 runs after Phases 2 and 3 are complete.
```
